Verify the attached note at all altitudes: proof structure, result consistency, computation, and a targeted deep check. Apply graduate-level mathematical rigor and assume nothing is correct until you have checked it.

This prompt is tuned for high recall: err on the side of flagging anything you cannot verify, even if you suspect the issue is spurious. The user expects false positives and is responsible for adjudicating them. Do not suppress a flag because you think it is probably fine.

Read the source AND any attached photos AND any reference PDFs deeply enough to verify the substance — every hypothesis, every step, every computation, every appeal to a named theorem. Photos may contain handwritten derivations to verify or margin annotations the author already flagged for self-verification ("is this right?", "?", `% CHECK`, circled equations); treat photo content as part of the same proof to be verified. If a User Background page is present, read it too — the user's research area may indicate which conventions they take for granted (and which the proof is implicitly leaning on).

Inline references policy. When a flag in any of the sections below is genuinely supported by the literature — a known counterexample, a textbook chapter that develops the same proof more carefully, a survey that resolves the disputed convention — cite the reference inline within the flag, in parentheses or as a trailing sentence (e.g., `[! the convention here disagrees with Royden §4.3]`). Do not bias toward recent work — classical sources by Cauchy, Riemann, Hadamard, Banach, etc. are first-class citizens; in mathematics the seminal reference is often decades or a century old. If you cannot verify the exact citation, either omit it or state your uncertainty in prose ("I recall a counterexample by Zygmund along these lines but cannot verify the source — please check before citing"). Hard rule: do not invent citations. One fabricated reference erodes trust in the whole verification report.

Section 1 — Adversarial review (proof-structure audit).
Read the note as a skeptical referee. If the note has a clear main result, audit the proof of that; if the note has multiple parallel claims, audit the most fully-stated one and briefly note which others were skipped. Check three things: (i) whether every hypothesis of the main result is actually used in the proof; (ii) whether the proof is complete — no skipped cases, no "similarly", no implicit appeals to results not stated; (iii) whether the proof relies on assumptions implicit in the author's mental model but not written down. For each issue, identify the location, state the issue, and explain in 1–2 sentences why it matters.

Section 2 — Counterexample search (result consistency).
Try to construct an object that satisfies all of the note's stated hypotheses but violates the conclusion. If you find one, the note contains an error: a hypothesis is missing, the conclusion is mis-stated, or there is a mistake in the proof. Report the counterexample explicitly, list all assumptions used in constructing it (including any concrete choices you committed to beyond the note's stated hypotheses), verify that it satisfies every stated hypothesis, exhibit how the conclusion fails, and identify which of the three error types it points to.

If after honest effort you cannot construct such a counterexample, say so plainly. Then briefly describe the classes of candidate objects you considered (e.g., constant or trivial cases, polynomial or Gaussian families, degenerate limits, pathological constructions in a particular regularity class) so the user can judge whether the search was thorough. Do not invent counterexamples that do not actually break the conclusion.

Section 3 — Computational sanity check.
Independently redo every non-trivial computation in the note — algebraic manipulations, integrations, applications of named theorems, limit calculations, sign tracking, index arithmetic. Include computations on photos (handwritten derivations on whiteboards, paper scans) — verify the math itself AND, if the photo content is also transcribed in source, verify the transcription is faithful. Do not copy the note's intermediate steps; compute from the inputs and compare your result to the note's. Flag discrepancies with [! ...] and explain in 1–2 sentences what looks suspect. Confirm matches briefly — one line per check, format "[line/equation reference]: verified". Do not rewrite the note's prose; this section is a checklist, not a rewrite.

If the note contains no non-trivial computations to verify (e.g., it is purely conceptual or structural), state that briefly in one sentence and skip the rest of the section rather than padding.

Section 4 — Targeted deep check.
Look in the note (source AND photos) for any step the author has flagged for verification — typically marked with `% CHECK`, `[?]`, "is this right?", "verify", or similar in source, or with a question mark / underline / circle in handwritten photo content. If you find such a flag, perform an extended verification of that specific claim: re-derive it from scratch by an independent route if possible, check edge cases, and explicitly compare to the note's argument. If the note contains no flag, identify the single step that looks most likely to contain an error and check that step in the same extended way. Briefly explain why you chose that step.

Section 5 — Reviewer's note to the user.
End the report with a short paragraph addressed to the user reminding them that this prompt is calibrated for high recall: many of the flags above are likely false positives, and the user should treat the report as a list of items to inspect, not a list of confirmed errors. Mention specifically that ambiguity in the source frequently produces a [! ...] flag even when the underlying mathematics is correct, and that flags in Section 1 (implicit assumptions) are particularly prone to this.

If a User Background page is present, also acknowledge that some flagged conventions may be standard in the user's subfield even though they look suspicious to a generic reader — direct the user to dismiss those quickly when they recognize them.
