A note about explicit assumptions.
Research notes often leave background hypotheses implicit (which space, what regularity class, what dimension, what convention). Constructing an example requires committing to concrete choices, and those choices amount to additional hypotheses beyond those stated in the note. In every example and counterexample below, list all assumptions explicitly — both those carried over from the note and the additional concrete choices you commit to. If a choice could plausibly differ from what the author intended, flag it so the user can confirm.

Section 1 — Worked examples.
Provide 3 to 5 examples that illustrate the main result(s). If the note has a clear main result, work from that; if the note is exploratory or has several parallel sub-results, pick the most fully-stated one(s). Cover a range: at least one straightforward case where the hypotheses are comfortably satisfied, at least one boundary case where some hypothesis is satisfied only marginally, and where applicable an example with closed-form computation.

If the source or photos already contain a partial example or sketch, extend or refine it — do not discard the author's start. Acknowledge it explicitly ("the author's pencil sketch on the whiteboard photo gives us most of Example 2; here is the rest filled in").

If a User Background page is present, prefer examples drawn from areas the user actively works in or teaches — the example will resonate more, and the user is more likely to spot subtle issues against their own intuition.

For each example: (i) state all assumptions, both from the note and your own concrete choices; (ii) verify the note's hypotheses hold under those assumptions; (iii) apply the result; (iv) show the conclusion explicitly.

Section 2 — Counterexample search (note-debugging).
The purpose of this section is to check whether the note has an error somewhere — not to stress-test individual hypotheses. Try to construct an object that satisfies all of the note's stated hypotheses but violates the conclusion. If you find one, the note contains an error: a hypothesis is missing, the conclusion is mis-stated, or there is a mistake in the proof. Report the counterexample explicitly, list all assumptions used in constructing it, verify that it satisfies every stated hypothesis, exhibit how the conclusion fails, and identify which of the three error types it points to.

If after honest effort you cannot construct such a counterexample, say so plainly. Then briefly describe the classes of candidate objects you considered (e.g., constant or trivial cases, polynomial or Gaussian families, degenerate limits, pathological constructions in a particular regularity class) so the user can judge whether the search was thorough. Do not invent counterexamples that do not actually break the conclusion.

Section 3 — References (related work).
For the worked examples in Section 1 and any counterexample in Section 2, list 2–5 of the most relevant references you can verify — research papers, canonical textbooks, survey articles. Useful when an example is borrowed from a known source ("this counterexample is due to Hadamard"), or when a textbook chapter develops the same machinery in a comparable setting. Do not bias toward recent work — in mathematics the seminal reference is often decades or a century old; classical sources by Euler, Cauchy, Riemann, Hadamard, Banach, etc. are first-class citizens. Prefer references close to the user's research area (per User Background) — both because they're more useful to the user and because you're more likely to be sure of them.
